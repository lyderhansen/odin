Fix the one real HARD-08 injection gap in `cron.sh`: line 158 emits `$script_name` and `$script` (filenames from `ls /etc/cron.daily/`) directly into the KV output without `safe_val`. An attacker who can write to `/etc/cron.daily/` could plant a file named `foo cron_extra=injected` and confuse the Splunk KV parser.

**Current (unsafe):**
```bash
emit "type=cron cron_source=cron.$period cron_schedule=@$period cron_command=$script_name cron_file=$script"
```

**Target (safe_val'd):**
```bash
emit "type=cron cron_source=cron.$period cron_schedule=@$period cron_command=$(safe_val "$script_name") cron_file=$(safe_val "$script")"
```

**Rationale for each wrap:**
- `$script_name` — basename output from `ls`, attacker-controlled if they can write to cron.daily/hourly/weekly/monthly dirs (requires root in normal setups, but defense in depth matters).
- `$script` — full path, same origin, same risk.

**Leave other fields alone:**
- `cron.$period` is a constant prefix with a literal interpolation from a fixed-domain variable (`daily|hourly|weekly|monthly`) — not attacker-controlled.
- `@$period` is the same fixed-domain value — safe.

**Preserve surrounding code.** This is a one-line surgical edit. Do not refactor the surrounding loop or change how `$script_name` is populated. The existing `safe_val()` function (defined at `cron.sh:28`) is already available in scope.
