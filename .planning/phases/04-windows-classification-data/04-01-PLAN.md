# Plan 04-01 — Windows Service + Port Classification Rows (Wave 0)

## Goal

Add Windows-specific rows to `odin_classify_services.csv` and `odin_classify_ports.csv` so that Windows hosts (DC, IIS, SQL Server, Exchange, DNS, DHCP, File, Print, Hyper-V, ADFS, RDS, SCCM, WSUS, ADCS, NPS, DFS, WDS, Failover Cluster, generic member, generic workstation — D2) classify to the existing cross-platform `host_role` taxonomy (D1) when the TA-ODIN orchestrator emits a `type=service` or `type=port` event.

## Why Wave 0

Service and port signals are the foundational data for host-role classification. They must land before Plan 02 (packages + log_sources), because:
- `odin_log_sources.csv` rows added in Plan 02 will reference `signal_type=service` and `signal_type=port` values that must already exist as classifiable signals
- AppInspect verification after Wave 0 confirms the foundational changes are clean before adding the more complex package patterns and log_source mappings in Wave 1

## Critical research findings (from 04-RESEARCH.md)

- **DO NOT modify `transforms.conf`** — the `match_type = WILDCARD(service_pattern)` line for `odin_classify_services` is already in place. CONTEXT.md D3 was based on a stale assumption.
- **Use Get-Service canonical names with case** per RESEARCH §3 (e.g., `MSSQLSERVER`, `W3SVC`, `vmms`). `case_sensitive_match = false` in transforms.conf forgives case mismatches but documentation case is the convention.
- **Use existing host_role values per D1** — `web_server`, `database_server`, `mail_server`, `domain_controller`, `dns_server`, `dhcp_server`, `file_server`, `print_server`, `virtualization_host`, `identity_server`, `certificate_server`, `rdp_server`, `management_server`, `ha_cluster`, `windows_management`, `windows_host`. Do NOT introduce `windows_iis`, `windows_dc`, etc.

## Out of scope for this plan

- Package patterns (Plan 02 owns `odin_classify_packages.csv`)
- Signal-to-log-source mapping (Plan 02 owns `odin_log_sources.csv`)
- Any changes outside `ODIN_app_for_splunk/lookups/`
- Modifying or removing existing Linux rows
- Adding `transforms.conf` changes (already in place; do not re-add)
