# v1.0.2 — Host Metadata Enrichment

## Goal

Every TA-ODIN scan emits ONE `type=odin_host_info` event with rich host
metadata, so Splunk dashboards and saved searches can classify, group, and
filter hosts by OS distribution, hardware profile, network identity, and
runtime environment — not just by `os=linux|windows`.

## The 13 fields (Variant B per /gsd-explore decision)

| Field | Example | Detection method (Linux) | Detection method (Windows) |
|---|---|---|---|
| `os_distro` | rocky, ubuntu, windows, alpine | parse `/etc/os-release` ID= | `Get-CimInstance Win32_OperatingSystem`.Caption (parsed) |
| `os_version` | 9.3, 22.04, 10.0.26100 | parse `/etc/os-release` VERSION_ID= | `[System.Environment]::OSVersion.Version` |
| `os_pretty` | "Rocky Linux 9.3 (Blue Onyx)" | parse `/etc/os-release` PRETTY_NAME= | Get-CimInstance.Caption |
| `os_kernel` | 5.14.0-362.el9.x86_64, 10.0.26100.4349 | `uname -r` | (Get-CimInstance Win32_OperatingSystem).BuildNumber |
| `os_arch` | x86_64, aarch64, amd64 | `uname -m` | `$env:PROCESSOR_ARCHITECTURE` |
| `cpu_cores` | 8 | `nproc` | (Get-CimInstance Win32_Processor).NumberOfCores |
| `mem_total_mb` | 16384 | `awk '/MemTotal/{print int($2/1024)}' /proc/meminfo` | (Get-CimInstance Win32_OperatingSystem).TotalVisibleMemorySize / 1024 |
| `uptime_seconds` | 432000 | `awk '{print int($1)}' /proc/uptime` | (Get-CimInstance Win32_OperatingSystem).LastBootUpTime → delta |
| `fqdn` | web01.prod.example.com | `hostname -f` | [System.Net.Dns]::GetHostByName($env:COMPUTERNAME).HostName |
| `ip_primary` | 10.0.5.123 | `ip route get 1.1.1.1 \| awk '{print $7;exit}'` | (Get-NetRoute -DestinationPrefix '0.0.0.0/0' \| Get-NetIPAddress).IPAddress |
| `virtualization` | kvm, vmware, hyperv, container, baremetal | `systemd-detect-virt` (or fallback dmidecode + cgroup probe) | (Get-CimInstance Win32_ComputerSystem).Manufacturer/Model |
| `cloud_provider` | aws, gcp, azure, none | curl IMDS endpoint with 1s timeout | Invoke-RestMethod IMDS endpoint with 1s timeout |
| `cloud_region` | eu-north-1, europe-west1, eastus | IMDS-specific endpoint | IMDS-specific endpoint |

## Event shape

ONE event per orchestrator run. Emitted between `type=odin_start` and the
first module event. Uses standard envelope:

```
timestamp=<ISO8601Z> hostname=<fqdn> os=<linux|windows> run_id=<id> odin_version=<ver>
type=odin_host_info os_distro=rocky os_version=9.3 os_pretty="Rocky Linux 9.3 (Blue Onyx)"
os_kernel=5.14.0-362.13.1.el9_3.x86_64 os_arch=x86_64
cpu_cores=8 mem_total_mb=16384 uptime_seconds=432000
fqdn=web01.prod.example.com ip_primary=10.0.5.123
virtualization=kvm cloud_provider=aws cloud_region=eu-north-1
```

Splunk saved-searches join this with module events using `run_id` (already
common to every event in a run).

## Why a separate event vs. extending odin_start

Decision deferred to v1.0.2 plan-phase, but leaning toward separate
`type=odin_host_info` because:
- Keeps `odin_start` semantically clean ("scan started, here's auth context")
- Allows host_info to grow over time without bloating start event
- Easier to gate (e.g., on hosts where IMDS times out, host_info can be
  emitted with partial fields without affecting odin_start latency)

## Acceptance criteria (draft)

1. Every Linux scan emits exactly one `type=odin_host_info` event with all 13 fields
2. Every Windows scan emits exactly one `type=odin_host_info` event with all 13 fields
3. Cross-platform parity test: Linux + Windows produce identical field-set
   (modulo platform-specific values like os_arch=x86_64 vs amd64)
4. Cloud IMDS probe times out cleanly within 2s on hosts WITHOUT cloud
   metadata (no false positives, no hangs)
5. Container detection: `virtualization=container` emitted when running inside
   docker/podman/k8s pod (preview of v1.1.0 Nivå 2 work)
6. DATA-DICTIONARY.md updated with full `type=odin_host_info` field reference
7. New dashboard panel: "OS Distribution" pie chart on odin_overview.xml
8. AppInspect baseline preserved (failure=0, error=0, warning=1)
9. All v1.0.1 regression tests still pass (HARD-01, PROD-01, PROD-05, windows-parity-harness)

## Risks (low overall)

- **IMDS endpoint behavior:** Cloud probing must be timeout-safe. Worst case:
  hostname-only event with no cloud_provider field. Mitigation: 1-2s timeout
  per probe, default to `cloud_provider=none` on any failure.
- **systemd-detect-virt unavailable:** On older Linux distros (RHEL 6, CentOS 6).
  Mitigation: fallback chain via dmidecode → /proc/cpuinfo → /proc/1/cgroup.
- **Windows ARM64:** Newer arch, may report PROCESSOR_ARCHITECTURE differently
  than expected. Mitigation: empirical test on Windows 11 ARM VM if available.

## NOT in scope for v1.0.2

These belong to v1.1.0 or later:
- Container enumeration (the actual containers running on a host) — v1.1.0 Nivå 3
- Cloud asset metadata (AWS Tags, GCP Labels, Azure tags) — v1.1.0 Phase 4 (optional)
- Network topology (default route, DNS servers, gateway) — defer to v1.2+
- Hardware inventory (PCI devices, GPUs, NVMe) — defer to v1.2+
- BIOS/firmware versions — defer to v1.2+ (security-audit territory)

## Building block for v1.1.0

v1.1.0 container observability needs to know:
- Is THIS host a container_host? (`virtualization` field hints + container runtime services)
- Is THIS host running INSIDE a container? (`virtualization=container` from this event)
- What OS base does the container have? (matches `os_distro` + `os_version`)

All three questions are answered by `type=odin_host_info` once v1.0.2 ships.
