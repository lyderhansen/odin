# v1.0.3 — Classification Accuracy (Role vs Feature Distinction)

## Goal

Fix host classification false-positives by distinguishing between host ROLES (what the machine's purpose IS) and host FEATURES (what the machine HAS installed by default). A blank Windows 11 workstation currently gets classified as domain_controller + file_server + print_server + rdp_server + security_endpoint — only windows_host is correct. After this fix, it should only be classified as windows_host.

## The problem (confirmed via Splunk data 2026-05-07)

Query against a freshly-installed Windows 11 VM ("test" hostname) in Splunk:

```spl
index=odin_discovery sourcetype=odin:enumeration (type=service OR type=port OR type=package) hostname="test"
| eval signal_type=type, signal_value=case(type="service", service_name, type="port", listen_port."/".transport, type="package", package_name)
| dedup hostname, signal_type, signal_value
| lookup odin_log_sources signal_type, signal_value
| where isnotnull(host_role)
| stats values(signal_type) AS detected_by, values(signal_value) AS signals by hostname, host_role
```

**Result: 6 roles assigned, 3 are false-positives (50% error rate):**

| Role | Signal | Correct? | Why |
|---|---|---|---|
| domain_controller | Netlogon service | FALSE POSITIVE | Netlogon runs on ALL domain-joined Windows, not just DCs |
| file_server | LanmanServer + 445/tcp | FALSE POSITIVE | SMB/CIFS is default on all Windows hosts |
| print_server | Spooler service | FALSE POSITIVE | Print Spooler is default Windows service |
| rdp_server | TermService | FALSE POSITIVE | RDP is enabled by default on most Windows |
| security_endpoint | WinDefend | FEATURE, not role | Windows Defender is an OS feature, not a host purpose |
| windows_host | EventLog, Schedule, mpssvc, wuauserv | CORRECT | Baseline Windows services = correct role |

## Design direction (from /gsd-explore discussion)

### Core concept: role vs feature

**Role** = what the machine IS (its purpose). Assigned by DEFINITIVE signals — services/ports/packages that someone deliberately installed and configured. Examples: nginx (web_server), mysqld (database_server), NTDS (domain_controller).

**Feature** = what the machine HAS by default. Present on every host of that OS type as part of the base installation. Should NOT drive role classification. Examples: Spooler (Windows default), LanmanServer (Windows default), WinDefend (Windows default), sshd (Linux default on many distros).

**Thumb rule:** If the service/port was deliberately installed for a purpose → role. If it ships with the OS and runs by default → feature.

### Implementation approach: signal_category column

Add a `signal_category` column to `odin_log_sources.csv` with values:
- `role` — signal is definitive for the host_role it maps to
- `feature` — signal indicates an OS-level capability, not a host purpose

**What changes:**

1. **`odin_log_sources.csv`** — new column added. All 311 existing rows categorized as `role` or `feature`. Re-categorization focuses on Windows ambient signals first (highest false-positive impact) then Linux equivalents.

2. **`transforms.conf`** — lookup binding may need updating to expose new column at search time.

3. **`savedsearches.conf`** — the three nightly saved searches (Host Inventory, Log Source Details, TA Deployment Matrix) filter on `signal_category=role` when computing host_role. Feature signals still available for separate queries.

4. **Dashboard** — optional: add "Host Features" panel alongside existing role-based panels so SOC analysts can still see what capabilities each host has.

### Re-categorization examples (non-exhaustive)

**Windows signals to reclassify as `feature`:**
- Netlogon → feature (runs on all domain-joined, not DC-only)
- Spooler → feature (default Windows service)
- LanmanServer → feature (default SMB sharing)
- TermService → feature (RDP default enabled)
- WinDefend → feature (Defender is OS-integrated)
- mpssvc (Windows Firewall) → feature
- EventLog, Schedule, wuauserv → feature (but these correctly map to windows_host which IS a role)

**Linux signals to review:**
- sshd → role or feature? Most Linux hosts run sshd. If feature, many hosts lose their only signal. Decision: keep as role for `ssh_server` since sshd IS an accessible service that SOC cares about.
- cron → feature (crond runs on all Linux; it's not a "cron server")

**Signals that remain `role`:**
- NTDS → domain_controller (definitive — only on DCs)
- nginx, apache, httpd → web_server (deliberately installed)
- mysqld, postgresql, mongod → database_server (deliberately installed)
- dockerd, kubelet → container_host (deliberately configured)
- named, bind9 → dns_server (server-side DNS, not client)

### Multi-role hosts

A host CAN legitimately have multiple roles. A DC running DNS Server has both domain_controller (from NTDS) and dns_server (from DNS Server service). This is correct — both are definitive role-signals for deliberately-configured services.

### Edge cases to decide during plan-phase

- **sshd on Linux:** role or feature? Leans role — sshd is an accessible service with security implications. But on cloud VMs, sshd is pre-installed and default.
- **IIS on Windows:** role — IIS requires deliberate installation via Add Roles and Features.
- **Port 445 on Windows:** feature (SMB default) — but on a NAS appliance? Need to consider whether the same signal can be role on one host_role and feature on another. Probably not worth the complexity — keep as feature universally.
- **Combination signals:** Should some roles REQUIRE multiple signals? E.g., domain_controller only if NTDS + port 389 + port 88 (all listening, not just any one)? Adds complexity but reduces false-positives further. Decision: defer to v1.0.4 or discuss-phase.
