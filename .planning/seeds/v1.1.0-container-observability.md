# v1.1.0 — Container Observability

## Goal

Make TA-ODIN aware of containerized infrastructure: detect when running INSIDE
a container, enumerate containers running ON a docker/k8s host, and classify
container workloads by image so Splunk dashboards show container fleet
composition + recommend appropriate Splunk TAs.

## The two scenarios (per /gsd-explore decision)

| Scenario | What TA-ODIN sees | What it should emit |
|---|---|---|
| **TA-ODIN INSIDE a container** | Container's filesystem view, no host services | `type=odin_host_info virtualization=container container_runtime=<docker\|podman\|kubepods>` (already in v1.0.2 scope as preview) |
| **TA-ODIN ON a docker/k8s host** | Host services include `dockerd`/`kubelet`; access to `docker ps` / `kubectl get pods` | `type=container` events listing each running container with image, name, command, ports, host_role classification |

Together: every host gets one `type=odin_host_info` (v1.0.2) PLUS optional N
`type=container` events (v1.1.0) per scan. Splunk can join these by `run_id`
to build full host/container topology.

## Phase decomposition (draft — refine in plan-phase)

### Phase 1 — Container environment detection (Nivå 2)
**Effort:** 4-8h
**Scope:**
- Extend orchestrator (Linux + Windows) to enrich `type=odin_host_info` event
  with refined container_runtime detection (already partially scoped in v1.0.2)
- Add fields: `container_runtime`, `container_id` (from /proc/1/cpuset or
  Docker env), `container_image_hint` (from `/etc/os-release` IMAGE_ID if set)
- Update DATA-DICTIONARY.md with container-specific host_info fields
**Acceptance:** Running TA-ODIN inside a docker container produces
`virtualization=container container_runtime=docker container_id=<sha-prefix>`
on Linux; equivalent on Windows containers.

### Phase 2 — Container enumeration module (Nivå 3 core)
**Effort:** 1-2 days
**Scope:**
- New `bin/modules/containers.sh` (Linux) and `bin/modules/containers.ps1` (Windows)
- Detects available container runtimes: docker, podman, kubectl
- For each runtime, enumerates running containers/pods via API (no log scraping)
- Emits one `type=container` event per container with: `container_id`,
  `container_image`, `container_name`, `container_command`, `container_ports`,
  `container_runtime`, `container_state`, `container_started_at`
- Handles common edge cases: docker daemon unreachable, no permission to
  list pods (returns `type=odin_warning module=containers`)
**Acceptance:** On a host with docker installed and containers running,
emits one `type=container` per container. On hosts without container
runtime, emits `type=none_found module=containers`.

### Phase 3 — Image-based classification + saved search
**Effort:** 1-2 days
**Scope:**
- New lookup: `ODIN_app_for_splunk/lookups/odin_classify_container_images.csv`
  - Pattern-based image-name → host_role mapping
  - e.g., `nginx*,web_server,Splunk_TA_nginx,...`
  - e.g., `mysql/mysql-server:*,database_server,Splunk_TA_mysql,...`
- New transforms.conf entry binding image_pattern lookup to container events
- New saved search: `[ODIN - Container Inventory]`
  - Aggregates `type=container` events into per-host container count + role distribution
  - Outputs `odin_container_inventory.csv` (parallel to odin_host_inventory.csv)
- New dashboard panel on odin_overview.xml showing container density per host
**Acceptance:** Saved search produces inventory CSV; dashboard shows
"hosts with N containers, fleet image-role distribution".

### Phase 4 (OPTIONAL) — Cloud auto-discovery
**Effort:** 2-3 days
**Scope:**
- Extend host_info detection from v1.0.2 with cloud asset metadata
- AWS: query IMDSv2 for instance-id, instance-type, tags, IAM role, AZ
- GCP: query metadata.google.internal for project, zone, machine-type, labels
- Azure: query 169.254.169.254 for subscription, resource-group, vm-size, tags
- Add fields to `type=odin_host_info`: `cloud_instance_id`, `cloud_instance_type`,
  `cloud_tags`, `cloud_iam_role`, `cloud_az`
- New dashboard panel showing fleet by cloud provider/region/instance type
**Acceptance:** On AWS/GCP/Azure VMs, `cloud_provider` and metadata fields
populated within 2s probe timeout. On non-cloud hosts, no errors, fields
remain unset.

**Why optional:** Cloud detection is a parallel track that doesn't depend
on container work. Could ship as v1.1.0 if scope holds, OR slip to v1.1.1
without blocking container observability.

## Architectural decisions to lock during plan-phase

| Decision | Options | Default leaning |
|---|---|---|
| Container API access | Direct binary (`docker ps`) vs unix-socket (`/var/run/docker.sock`) vs HTTP API | Direct binary — simpler, no auth, leverages existing CLI permissions |
| K8s API access | `kubectl get pods` vs k8s API client | `kubectl` — same simplicity argument; assumes kubectl is on PATH and ~/.kube/config valid |
| Permission model | Run as root vs leverage docker group | Document both; root for production deployments, docker-group for dev |
| Pod metadata depth | Just running pods vs pods + nodes + services + ingress | Just pods for v1.1.0; add cluster-topology in later milestone |
| Image registry resolution | Just image name vs resolve to digest+manifest | Just name for v1.1.0; digest resolution requires registry auth (later) |

## Cross-cutting risks

- **k8s RBAC complexity:** Production k8s clusters have RBAC that may deny
  TA-ODIN access to list pods. Mitigation: ship with documented RBAC manifest
  (ClusterRole + ClusterRoleBinding) that operators can apply.
- **Container churn:** Containers can be created/destroyed within a single scan
  cycle. `type=container` event represents snapshot at scan time. Splunk
  dashboards must explain this lifecycle constraint.
- **Image name diversity:** `nginx`, `nginx:latest`, `nginx:1.25-alpine`,
  `docker.io/library/nginx:1.25-alpine` all refer to nginx. Lookup needs
  pattern-matching tolerance. Mitigation: use Splunk lookup WILDCARD match_type
  (already in transforms.conf for packages — proven pattern).
- **Windows containers:** Different runtime model than Linux containers.
  May need separate Windows containers module or significant logic split.
  Mitigation: validate Windows container test environment exists BEFORE
  starting Phase 2; if not, defer Windows containers to v1.1.1.

## NOT in scope for v1.1.0

- Service mesh observability (Istio, Linkerd, Consul Connect)
- Container security scanning (Trivy, Snyk, Aqua) integration
- Container registry inventory (what's in your registry vs what's running)
- Container resource utilization metrics (CPU/memory per container)
- Build pipeline metadata (which CI built this image)
- Kubernetes cluster topology beyond pods (services, ingress, networkpolicies)
- Helm release inventory

These could become v1.2+ candidates. Cluster topology is the most likely
v1.1.1 follow-on if v1.1.0 is well-received.

## Testing strategy

- **Phase 1:** Existing windows-parity-harness extended with container-env probes
- **Phase 2:** New `tools/tests/check-container-enumeration.sh` regression guard
  - Validates `bin/modules/containers.sh` exits 0 on host without docker
    (returns `type=none_found module=containers`)
  - Validates fields-emitted contract on host with docker (using a known
    fixture image like `alpine:latest`)
- **Phase 3:** Saved-search test against synthetic container events
  (parallel to existing PROD-01 windows-classification regression)
